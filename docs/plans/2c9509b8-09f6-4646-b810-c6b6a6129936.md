## Context

We are scaffolding a hackathon-grade Web3 Smart RPC Router. The broader system targets a local-only, cyberpunk-styled TUI dashboard that fronts multiple free Web3 RPC endpoints and provides transparent failover against `429` and `5xx` responses. Phase 1 has no business logic — it exists to lock down the configuration contract (declared declaratively in YAML) behind a Pydantic v2 schema, with comprehensive negative tests. The repository is currently empty save for `.gitignore` and `.humanize/`. There is no legacy code or pattern to match; the implementer is greenfield.

## Assumptions

- The user is on a Linux container with Python 3.11+ available on `PATH` (the harness guarantees this).
- `pip` is available for installing `requirements.txt`; no Conda/Poetry is required.
- Pydantic v2 (`>=2.6`) is acceptable and preferred over v1. The `pydantic` package name remains the same.
- `routing_strategy` allowed values are locked to: `round_robin`, `priority`, `lowest_latency`, `failover` (extensible later, but Phase 1 rejects unknown values).
- The hackathon scoring treats **strict config rejection** as a positive signal: invalid YAML, missing fields, malformed URLs, and out-of-range numbers MUST raise `pydantic.ValidationError`, NOT fall back to defaults.
- All configuration knobs referenced in the Phase 1 spec — `global.listen_port`, `global.probe_interval`, `rpc_nodes[].provider`, `rpc_nodes[].url`, `rpc_nodes[].routing_strategy`, `rpc_nodes[].priority` — are required unless an explicit default is documented in the schema.
- HTTP is acceptable for `rpc_nodes[].url` (some public testnets serve plain HTTP), but `file://` and missing scheme MUST be rejected.

## Acceptance Criteria

- AC-1: Repository contains the directory layout above with empty `__init__.py` files in `core/`, `ui/`, and `tests/`.
- AC-2: `requirements.txt` pins `pydantic>=2.6,<3`, `PyYAML>=6.0`, `pytest>=8.0`, `pytest-asyncio>=0.23`, `pytest-cov>=4.1`, and dev tools (`mypy`, `ruff`).
- AC-3: `core/models.py` defines the following Pydantic v2 models with field-level constraints and a `model_config = ConfigDict(extra="forbid")`:
  - `GlobalSettings`: `listen_port: int` (1–65535), `probe_interval_seconds: float` (>0), `request_timeout_seconds: float` (>0), `max_retries: int` (>=1, default 3).
  - `RoutingStrategy` (string enum): `ROUND_ROBIN`, `PRIORITY`, `LOWEST_LATENCY`, `FAILOVER`.
  - `RpcNode`: `provider: str` (non-empty, stripped), `url: AnyHttpUrl`-compatible validator that rejects non-`http(s)` schemes, `routing_strategy: RoutingStrategy`, `priority: int` (>=1, unique within config), optional `weight: int` (>=1, default 1), optional `headers: dict[str, str]` (default `{}`).
  - `RouterConfig`: `global_: GlobalSettings` aliased to YAML key `global`, `rpc_nodes: list[RpcNode]` (min length 1).
  - Each `RpcNode.provider` value MUST be unique across `rpc_nodes`.
  - `model_validator` on `RouterConfig` enforces unique `provider` AND unique `priority` across the list.
- AC-4: `core/config.py` exposes exactly two public functions:
  - `load_config(path: str | Path) -> RouterConfig` — reads YAML, converts to dict, instantiates `RouterConfig(**raw)` so all Pydantic errors surface naturally.
  - `parse_config_dict(raw: dict) -> RouterConfig` — pure function used by tests and by `load_config`.
  - Both functions raise `pydantic.ValidationError` (or its YAML-loading subclass) on bad input; they do NOT swallow errors.
- AC-5: `config.yaml` at repo root contains a working sample with two nodes (one HTTP, one HTTPS), a `listen_port: 8545`, `probe_interval_seconds: 5.0`, and uses `priority` routing. Loading it via `load_config("config.yaml")` MUST succeed.

  - `probe_interval_seconds` <= 0 raises `ValidationError`.

  - Duplicate `provider` across nodes raises `ValidationError` (model_validator).
